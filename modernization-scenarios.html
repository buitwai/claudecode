Scenario 1: Inventory Service Extraction

Intermediate 8-10 hours

Background

AutoZone's inventory management is deeply embedded in the AS/400 mainframe. To enable modern capabilities like real-time updates and mobile access, we need to extract inventory operations into a dedicated microservice.

Your Mission

Design and implement a modern inventory microservice that gradually replaces mainframe functionality while maintaining data consistency and system reliability.

Migration Strategy Requirements

  • Strangler Fig pattern for gradual migration
  • Dual-write strategy for data synchronization
  • Event-driven architecture for real-time updates
  • API-first design for cross-platform access
  • Rollback capability in case of issues
  • Comprehensive monitoring and logging

Claude Code Prompt Example

Extract inventory management from AutoZone's AS/400 mainframe:

Current state:
- All inventory operations handled by COBOL programs
- Batch updates every 15 minutes
- No real-time capabilities
- Tightly coupled with other business logic

Target microservice:
- Real-time inventory tracking
- RESTful APIs for all operations
- Event streaming for updates
- Independent deployment and scaling

Implementation approach:
1. Create new inventory service with PostgreSQL
2. Implement dual-write to mainframe and new service
3. Gradually route read operations to new service
4. Migrate write operations after confidence builds
5. Decomission mainframe inventory components

Include database schema design, API specifications, event schemas, and migration scripts. Handle edge cases like network failures and data inconsistencies.

Learning Objectives

  • Strangler Fig migration pattern
  • Event-driven architecture design
  • Data synchronization strategies
  • Legacy system integration techniques

Scenario 3: Distributed Transaction Management

Advanced 8-10 hours

Background

Order processing in AutoZone involves multiple steps: inventory reservation, payment processing, shipping allocation, and customer notification. In a microservices architecture, this requires sophisticated transaction management.

Your Mission

Implement distributed transaction patterns to ensure consistency across microservices without traditional ACID transactions.

Claude Code Prompt Example

Implement distributed transactions for AutoZone order processing:

Order flow across services:
1. Order Service: Create order record
2. Inventory Service: Reserve items
3. Payment Service: Process payment
4. Shipping Service: Allocate fulfillment
5. Notification Service: Confirm to customer

Challenges:
- What if payment fails after inventory is reserved?
- How to handle partial inventory availability?
- Network failures between service calls
- Ensuring eventual consistency

Patterns to implement:
- Saga pattern with orchestration
- Compensation actions for rollback
- Event sourcing for audit trail
- Circuit breakers for resilience
- Idempotency for safe retries

Build saga orchestrator with comprehensive error handling and monitoring for complex business transactions.

Migration Patterns & Best Practices

Strangler Fig Pattern

Gradually replace legacy functionality by intercepting calls and routing to new services

Legacy → Proxy → New Service

Database-First Migration

Migrate data layer first, then build new applications on modern data platform

Data Migration → API Layer → Frontend

API-First Approach

Create modern APIs wrapping legacy systems before building new interfaces

Legacy → API Gateway → Modern Apps

Event-Driven Transition

Use events to synchronize data between legacy and modern systems during migration

Legacy → Events → New Services
